# Процесс разработки архитектуры SDK

Полное руководство по проведению архитектурной работы для создания SDK.
От сбора требований до валидированной архитектуры.

---

## Модель итерации

Процесс **НЕ линейный**. Каждая последующая фаза может инвалидировать результаты предыдущих.

### Checkpoints между фазами

| Переход | Checkpoint | Если не проходит |
|---------|-----------|-----------------|
| 0 → 1 | Все must-have требования имеют acceptance criteria | Вернуться в 0.2, задать вопросы |
| 1 → 2 | Карта сущностей полна, все экспорты документированы | Перечитать файлы |
| 2 → 3 | Для каждого must-have use case есть решение хотя бы у 1 аналога | Расширить список аналогов или пересмотреть требование |
| 3 → 4 | Все must-have use cases маппятся на все бэкенды | Вернуться в Фазу 0 — пересмотреть приоритеты (must→should) или в Фазу 3 — найти эмуляцию |
| 4 → 5 | Все интерфейсы покрывают must-have use cases | Вернуться в 4.4, пересмотреть интерфейсы |
| 5 → 6 | Все BLOCKING issues исправлены, прототип работает | Итерировать в рамках Фазы 4-5 |

### Правила rework

1. **Инвалидация поздней фазой** — если Фаза N обнаруживает проблему в Фазе M (N > M):
   - Обновить артефакт Фазы M
   - Проверить все промежуточные фазы M+1..N-1 — не затронуты ли
   - Пройти checkpoint перед повторным входом в Фазу N

2. **Cascading updates** — какие артефакты зависят друг от друга:
   - REQUIREMENTS.md → USE-CASES.md → README-DRAFT.md → интерфейсы → примеры
   - Изменение требования → пересмотреть всю цепочку вправо

3. **ADR при откате** — каждый откат документируется как Architecture Decision Record:
   - Что пробовали, почему не сработало, что решили вместо этого

## Фаза 0: Сбор и формализация требований

### 0.1 Извлечение требований из задачи

**Входные данные:** описание задачи от стейкхолдера (часто неструктурированное, на естественном языке).

**Действия:**
1. Выписать ВСЕ упомянутые функции/возможности — дословно, без интерпретации
2. Выписать ВСЕ ограничения ("должно быть опционально", "не дублировать", "можно ломать API")
3. Выписать ВСЕ упомянутые проекты/системы/технологии
4. Идентифицировать неявные требования (что подразумевается но не сказано)
5. Составить список вопросов по каждому неоднозначному пункту

**Артефакт:** `REQUIREMENTS-RAW.md` — сырые требования с категоризацией:
- Функциональные (что должно делать)
- Нефункциональные (как должно работать: производительность, удобство API, extensibility)
- Ограничения (что НЕ должно делать, что опционально)
- Неопределённости (вопросы к стейкхолдеру)

### 0.2 Прояснение требований

**Действия:**
1. Задать все вопросы из 0.1 стейкхолдеру
2. Для каждого ответа — обновить REQUIREMENTS-RAW.md
3. Повторять пока нет неопределённостей уровня "блокирует дизайн"

**Критерий выхода:** ВСЕ вопросы уровня "блокирует дизайн" получили ответы. Допустимо оставить "nice-to-have" вопросы открытыми.

### 0.3 Структуризация требований

**Действия:**
1. Преобразовать сырые требования в user stories формата:
   "Как [роль] я хочу [действие] чтобы [ценность]"
2. Для каждой user story определить acceptance criteria
3. Группировать по доменам (sessions, storage, events, UI components, errors)
4. Приоритизировать: Must-have / Should-have / Nice-to-have (MoSCoW)

**Артефакт:** `REQUIREMENTS.md` — структурированные требования с приоритетами.

### 0.4 Определение потребителей и их use cases

**Действия:**
1. Идентифицировать ВСЕ типы потребителей SDK:
   - Разработчик простого чат-бота
   - Разработчик enterprise-приложения с множеством чатов
   - Разработчик собственного UI фреймворка
   - Другой SDK/библиотека использующая наш SDK
2. Для каждого типа потребителя написать 3-5 конкретных сценариев использования
3. Для каждого сценария написать псевдокод "идеального API" — как потребитель ХОЧЕТ писать код

**Артефакт:** `USE-CASES.md` — сценарии с идеальным API (README-driven design).

## Фаза 1: Анализ существующего кода

### 1.1 Инвентаризация текущего SDK

**Действия:**
1. Прочитать ВСЕ исходники — каждый файл, каждый экспорт
2. Составить полную карту сущностей:
   - Имя, тип (interface/class/type), файл:строка
   - Ответственность (одним предложением)
   - Зависимости (что использует)
   - Зависящие (кто использует)
3. Нарисовать граф зависимостей
4. Подсчитать метрики: LOC, количество экспортов, глубина зависимостей

**Артефакт:** `ENTITY-MAP.md` — полная карта с графом зависимостей.

### 1.2 Анализ проблем текущего SDK

**Действия:**
1. Для каждой сущности оценить:
   - Нарушение SRP (Single Responsibility)?
   - Дублирование с другой сущностью?
   - Отсутствие нужной функции?
   - Неудобный API для потребителя?
   - Несовместимость с требованиями из Фазы 0?
2. Классифицировать: CRITICAL / MAJOR / MINOR
3. Для каждой проблемы — конкретная ссылка (файл:строка)

**Артефакт:** Секция "Issues" в `ARCHITECTURE-REVIEW.md`.

### 1.3 Анализ реальных потребителей

**Действия:**
1. Найти ВСЕ проекты использующие SDK
2. Для каждого проекта:
   - Какие части SDK используют
   - Какие workarounds/костыли применяют
   - Какие типы дублируют
   - Какие баги/issues документировали
   - Какие паттерны повторяются в нескольких проектах
3. Собрать общие pain points

**Артефакт:** Секция "Consumer Analysis" в `ARCHITECTURE-REVIEW.md`.

## Фаза 2: Исследование аналогов

### 2.1 Определение конкурентов для анализа

**Действия:**
1. Составить список SDK/библиотек решающих аналогичную задачу
2. Для каждого определить:
   - Позиционирование (low-level / high-level / full-stack)
   - Целевая аудитория
   - Размер и зрелость (stars, contributors, release cycle)
3. Выбрать 4-6 наиболее релевантных для глубокого анализа

### 2.2 Глубокий анализ каждого аналога

**Действия для каждого SDK:**
1. Клонировать исходники (не документация — ИСХОДНИКИ)
2. Найти и изучить:
   - Core types/interfaces — публичный API surface
   - Архитектурные паттерны (layered, event-driven, transport-based)
   - Как решена каждая задача из наших требований (Фаза 0)
   - Что сделано хорошо (adoptable patterns)
   - Что сделано плохо или отсутствует (gaps)
3. Записать конкретные примеры кода из их исходников

**Критерий качества:** не "Vercel AI использует транспорт" а конкретный интерфейс ChatTransport с полями, файл, строка.

### 2.3 Сравнительный анализ

**Действия:**
1. Составить матрицу: наше требование × SDK → как решено
2. Выявить паттерны — что решается одинаково у всех
3. Выявить уникальные подходы — что один SDK делает иначе (и почему)
4. Определить industry best practices
5. Определить anti-patterns (что НЕ делать)

**Артефакт:** Секция "Research" в `ARCHITECTURE-REVIEW.md` со сравнительной матрицей.

## Фаза 3: Маппинг на бэкенды

### 3.1 Инвентаризация API бэкендов

**Действия:**
Для каждого бэкенда (Copilot SDK, Claude SDK, Vercel AI SDK):
1. Прочитать ВСЕ типы и методы из .d.ts файлов
2. Выписать полный API surface связанный с:
   - Создание/восстановление сессий
   - Отправка сообщений
   - Получение истории
   - Events/callbacks/hooks
   - Управление моделью
   - Permissions
3. Отметить ограничения каждого бэкенда (что НЕЛЬЗЯ сделать)

### 3.2 Таблица маппинга

**Действия:**
1. Для каждой нашей абстракции (из требований) определить:
   - Как маппится на бэкенд A (конкретный метод)
   - Как маппится на бэкенд B
   - Как маппится на бэкенд C
   - Что нужно эмулировать (бэкенд не поддерживает нативно)
2. Выявить операции невозможные на каком-то бэкенде
3. Определить lowest common denominator vs backend-specific features

**Артефакт:** Таблица маппинга в `ARCHITECTURE-REVIEW.md`.

## Фаза 4: Проектирование архитектуры

### 4.0 SDK Quality Attributes

**Действия:**
1. Определить бюджеты:
   - Max bundle size per entry point (KB gzipped)
   - Max количество peer dependencies
   - Max глубина зависимостей
2. Определить TypeScript constraints:
   - Target ES version
   - Module resolution strategy (ESM-first, CJS compat)
   - Generics strategy (explicit vs inferred)
   - Quality of IDE autocomplete (проверить в реальном IDE)
3. Определить extension points:
   - Какие интерфейсы потребитель может реализовать (storage adapters, event handlers)
   - Какие абстракции sealed (нельзя расширить)
   - Plugin/middleware model (если применимо)
4. Определить runtime requirements:
   - Minimum Node.js version
   - Browser compatibility (если есть UI компоненты)
   - Edge runtime compatibility (Vercel Edge, Cloudflare Workers)
5. Определить версионирование:
   - Semver policy
   - Deprecation strategy (минимум 1 minor до удаления)
   - Supported versions (N, N-1)

**Артефакт:** Секция "Quality Attributes" в `ARCHITECTURE-REVIEW.md`.

**Checkpoint:** Все constraints определены ДО начала проектирования интерфейсов. Они являются входными ограничениями для шагов 4.1-4.9.

### 4.1 User Journey mapping

**Действия:**
1. Для каждого типа потребителя (из 0.4) описать полный user journey:
   - Инициализация SDK
   - Создание первого чата
   - Отправка сообщения, получение стриминга
   - Показ tool calls, thinking blocks
   - Обработка permission requests
   - Обработка ошибок (network, rate limit, context overflow)
   - Переключение между чатами
   - Восстановление после рестарта приложения
   - Настройка кастомного storage
   - Подключение UI компонентов
2. Для каждого шага journey — определить какие API вызовы нужны
3. Нарисовать sequence diagrams для ключевых потоков

**Артефакт:** `USER-JOURNEYS.md` с диаграммами.

### 4.2 README-Driven Design

**Действия:**
1. Написать README.md SDK "из будущего" — как если бы SDK уже существовал:
   - Quick Start (5 строк до работающего чата)
   - Basic Usage (отправка, стриминг, history)
   - Sessions (создание, переключение, восстановление)
   - Events & Hooks (подписки на события, permissions)
   - Storage Adapters (InMemory, SQLite, custom)
   - UI Components (если есть)
   - Error Handling
   - Advanced (смена модели, смена адаптера, custom backends)
2. Каждая секция README — это примеры кода которые потребитель будет копировать
3. Примеры должны быть реалистичными — не "Hello World", а реальные задачи

**Критерий:** если потребитель не может скопировать пример и получить работающий код — пример плохой.

**Артефакт:** `README-DRAFT.md` — драфт README из будущего.

### 4.2.5 Architecture Decision Records (ADR)

**Действия:**
Для каждого ключевого архитектурного решения записать:
1. **Контекст** — какая проблема решается
2. **Рассмотренные варианты** — 2-3 альтернативы с описанием
3. **Решение** — что выбрано
4. **Обоснование** — почему именно это
5. **Последствия** — trade-offs, что теряем, что получаем
6. **Статус** — proposed / accepted / rejected / superseded

**Формат:** один ADR на одно решение. Нумерация: ADR-001, ADR-002, ...

**Примеры решений требующих ADR:**
- "Session vs Agent как основная сущность" (уже 3 попытки — v1/v2/v3)
- "Model привязан к сессии или передаётся per-call?"
- "Storage adapter обязательный или опциональный?"
- "Push vs Pull для событий стриминга"
- "Единый entry point vs separate entry points per backend"

**Артефакт:** `decisions/` директория или секция в `ARCHITECTURE-REVIEW.md`.

**Критерий:** если решение можно легко оспорить вопросом "а почему не X?" — нужен ADR.

### 4.3 Определение архитектурных слоёв

**Действия:**
1. Из user journeys и README вычленить слои SDK:
   - **Transport layer** — соединение с бэкендом (CLI subprocess, HTTP API)
   - **Session layer** — управление разговорами, контекстом, историей
   - **Event layer** — events, subscriptions, hooks, middleware
   - **Storage layer** — persistence, adapters, migration
   - **Error layer** — классификация, retry, user-facing errors
   - **UI layer** — компоненты, renderers (если есть)
2. Для каждого слоя определить:
   - Ответственность (что делает)
   - Публичный API (что видит потребитель)
   - Внутренний API (что видят другие слои)
   - Зависимости (какие слои использует)
3. Нарисовать layer diagram
4. Проверить: нет ли циклических зависимостей между слоями

**Артефакт:** Layer diagram + описание каждого слоя.

### 4.4 Определение интерфейсов

**Действия:**
1. Для каждого слоя определить ключевые интерфейсы:
   - Название, JSDoc описание
   - Методы с сигнатурами
   - Обоснование каждого метода (зачем нужен, какой use case покрывает)
2. Для каждого интерфейса проверить:
   - Следует ли SRP? (одна ответственность)
   - Можно ли расширить не ломая? (Open-Closed)
   - Зависит ли от абстракций, не от деталей? (DIP)
   - Не слишком ли много методов? (Interface Segregation)
3. Определить какие интерфейсы обязательны для потребителя, какие опциональны

**Артефакт:** TypeScript интерфейсы в `ARCHITECTURE-REVIEW.md`.

### 4.5 State machine design

**Действия:**
1. Для каждой stateful сущности (session, connection) определить:
   - Все возможные состояния
   - Все возможные переходы
   - Триггеры переходов
   - Side-effects при переходах
2. Нарисовать state diagram
3. Проверить:
   - Нет ли unreachable states
   - Нет ли deadlocks
   - Все ошибочные состояния обработаны

### 4.5.5 Streaming & Event Architecture

**Действия:**
1. **Event model** — определить подход:
   - Push (callbacks/EventEmitter) vs Pull (AsyncIterator) vs Hybrid
   - Для каждого подхода — pros/cons для данного SDK
   - Выбор с ADR
2. **Event normalization** — как 3 разных формата событий бэкендов маппятся на единый SDK event model:
   - Copilot: `assistant.message`, `tool.execution_start/complete`, `assistant.reasoning_delta`
   - Claude: `content_block_start/delta/stop`, `tool_use`, `thinking`
   - Vercel AI: `text-delta`, `tool-call`, `tool-result`, `step-finish`
   - Определить unified event set с маппингом на каждый бэкенд
3. **Streaming lifecycle** — полный flow:
   - `start → text_delta → tool_call_start → tool_call_end → text_delta → end`
   - С ветвлениями: thinking blocks, multiple tool calls, partial JSON
   - State diagram отдельно от session state machine
4. **Cancellation** — AbortSignal propagation:
   - User clicks "Stop" → SDK → CLI subprocess / HTTP request
   - Cleanup: partial events, incomplete tool calls
   - Graceful vs force abort
5. **Backpressure** — что если consumer медленнее producer:
   - Buffering strategy
   - Max buffer size
   - Overflow behavior (drop / block / error)
6. **Reconnection** — для persistent connections:
   - Auto-reconnect policy
   - State recovery after reconnect
   - Deduplication of events

**Артефакт:** Streaming state diagram + event normalization table.

### 4.6 Error taxonomy

**Действия:**
1. Составить полную таксономию ошибок:
   - Network errors (connection, DNS, timeout)
   - Auth errors (expired, invalid, forbidden)
   - Rate limit errors (with retry-after)
   - Provider errors (5xx, model unavailable)
   - Validation errors (invalid input, schema mismatch)
   - Context errors (overflow, token limit)
   - Permission errors (denied by user/policy)
   - SDK internal errors (bug, misconfiguration)
2. Для каждого типа ошибки определить:
   - Retryable? (да/нет)
   - User-facing message
   - Recovery strategy (auto-retry, ask user, abort)
   - HTTP status mapping (если применимо)

### 4.7 Backward compatibility analysis

**Действия:**
1. Выписать ВСЕ текущие публичные API
2. Для каждого определить: сохраняется / меняется / удаляется
3. Для каждого изменения написать migration guide
4. Определить можно ли сделать постепенную миграцию (deprecation) или только breaking change

**Артефакт:** Breaking changes таблица с migration paths.

### 4.8 Security & Trust Model

**Действия:**
1. **Trust boundaries** — определить границы доверия:
   - SDK ↔ бэкенд (CLI subprocess / API)
   - SDK ↔ приложение потребителя
   - SDK ↔ storage adapter
   - SDK ↔ tool execution environment
2. **Threat modeling** (STRIDE или аналог) для каждого слоя:
   - Spoofing: может ли вредоносный tool выдать себя за другой?
   - Tampering: могут ли tool parameters быть модифицированы?
   - Repudiation: можно ли отрицать что tool был вызван?
   - Information disclosure: утечка токенов через storage/logs?
   - Denial of service: бесконечный стриминг, resource exhaustion?
   - Elevation of privilege: permission bypass через SDK?
3. **Credential policy** — как SDK обращается с auth tokens:
   - Хранение: "SDK не хранит токены" (текущая политика) — документировать как ADR
   - Передача: в каких API calls передаются, как протектятся
   - Lifecycle: rotation, expiration, refresh
4. **Permission escalation** — сценарии:
   - Tool запрашивает shell access → пользователь разрешает → tool получает доступ ко всему
   - Как ограничить blast radius?
   - Связь с permission scopes (once / session / project / always)

**Артефакт:** Security model diagram + threat table.

### 4.9 Testing Architecture

**Действия:**
1. **Test boundaries per layer:**
   - Unit: каждый слой изолированно, все зависимости замоканы
   - Integration: слои вместе, бэкенды замоканы
   - E2E: реальные CLI бэкенды (требует auth)
2. **Mock strategy per backend:**
   - Copilot: `_injectSDK()` с mock client → mock event stream
   - Claude: `_injectSDK()` с mock query function → mock async iterator
   - Vercel AI: mock HTTP server или mock `generateText`/`streamText`
   - Определить: dependency injection через конструктор vs test-only methods
3. **Streaming test approach:**
   - Как тестировать partial events?
   - Как тестировать cancellation mid-stream?
   - Как тестировать backpressure?
   - Helpers: `createMockStream()`, `collectEvents()`, `assertEventSequence()`
4. **Error scenario testing:**
   - Network failure injection
   - Rate limit simulation
   - Auth expiration mid-session
   - Context overflow
5. **CI pipeline:**
   - Unit + integration: каждый PR
   - E2E: nightly / manual (requires real CLI auth)
   - Матрица: Node.js versions × backends

**Артефакт:** Testing strategy в `ARCHITECTURE-REVIEW.md`.

## Фаза 5: Валидация архитектуры

### 5.1 Проверка против требований

**Действия:**
1. Для КАЖДОГО требования из Фазы 0 — проверить что архитектура его покрывает
2. Формат: Требование → Как покрыто → Конкретный интерфейс/метод
3. Если требование НЕ покрыто — это BLOCKING issue

### 5.2 Проверка user journeys

**Действия:**
1. Для КАЖДОГО user journey из 4.1 — "проиграть" его через новые интерфейсы
2. Написать полный код (не псевдокод) для каждого journey
3. Проверить: не нужно ли потребителю делать неинтуитивные вещи?
4. Проверить: сколько строк кода нужно для каждого journey?

### 5.3 Проверка edge cases

**Действия:**
1. Для каждой ключевой операции (create session, send message, restore) определить edge cases:
   - Что если параллельные вызовы?
   - Что если сеть упала mid-stream?
   - Что если storage недоступен?
   - Что если бэкенд сменился mid-session?
   - Что если модель не поддерживает structured output?
2. Для каждого edge case — как архитектура его обрабатывает

### 5.4 Независимый ревью

**Действия:**
1. Делегировать ревью независимому субагенту (или человеку)
2. Ревьювер проверяет:
   - Полноту покрытия требований
   - Consistency интерфейсов
   - SOLID compliance
   - Backward compatibility
   - Реалистичность примеров
3. Исправить все BLOCKING и MAJOR issues
4. Повторить ревью если были BLOCKING issues

### 5.5 Prototype validation

**Действия:**
1. Написать минимальный работающий прототип (один бэкенд, основные операции)
2. Написать демо-приложение использующее прототип
3. Проверить на практике: удобен ли API?
4. Итерировать дизайн на основе обнаруженных неудобств

## Антипаттерны (что НЕ делать)

1. **Не начинать с интерфейсов** — начинать с use cases и README
2. **Не читать документацию вместо кода** — документация врёт, код не врёт
3. **Не проектировать "снизу вверх"** — от бэкенда к потребителю. Только "сверху вниз" — от потребителя к бэкенду
4. **Не оптимизировать количество интерфейсов** — "4 не более" — искусственное ограничение. Нужно столько сколько нужно
5. **Не копировать архитектуру конкурента** — извлекать паттерны, не копировать решения
6. **Не игнорировать edge cases** — "потом разберёмся" = баги в продакшене
7. **Не смешивать "что" и "как"** — интерфейсы = "что", реализация = "как". Сначала "что"
8. **Не забывать про DX** — Developer Experience = количество строк кода для типичной задачи. Меньше = лучше
